# Context Gatherer Agent - API Documentation

## Описание

Специализированный агент для эффективного исследования структуры репозитория и сбора релевантного контекста. Анализирует кодовую базу для идентификации файлов и секций кода, необходимых для решения проблемы.

---

## API Вызова

### Метод вызова
```
invokeSubAgent
```

### Обязательные параметры

#### name
- **Тип**: `string`
- **Значение**: `"context-gatherer"`
- **Описание**: Идентификатор агента

#### prompt
- **Тип**: `string`
- **Описание**: Описание проблемы или области для анализа
- **Формат**: Вопрос или описание того, что нужно понять в кодовой базе

#### explanation
- **Тип**: `string`
- **Описание**: Объяснение цели сбора контекста
- **Формат**: Краткое обоснование необходимости анализа

### Опциональные параметры

#### preset
- **Тип**: `string | null`
- **Значение**: `null` (не используется)

---

## Входные данные

### Структура prompt

Prompt должен описывать:
- Проблему или вопрос, требующий понимания кодовой базы
- Область интереса (если известна)
- Что именно нужно понять или найти
- Контекст проблемы (баг, новая функция, рефакторинг)

### Примеры входных данных

#### Пример 1: Исследование бага
```json
{
  "name": "context-gatherer",
  "prompt": "Пользователи сообщают, что форма регистрации не отправляет данные на сервер. Нужно понять, какие компоненты и файлы отвечают за обработку формы регистрации.",
  "explanation": "Сбор контекста для расследования бага с формой регистрации"
}
```

#### Пример 2: Понимание архитектуры
```json
{
  "name": "context-gatherer",
  "prompt": "Как реализована система аутентификации в этом проекте? Какие файлы отвечают за login, token management, и session handling?",
  "explanation": "Анализ архитектуры системы аутентификации перед добавлением новой функциональности"
}
```

#### Пример 3: Поиск зависимостей
```json
{
  "name": "context-gatherer",
  "prompt": "Найди все места, где используется функция calculateDiscount, и как она взаимодействует с другими компонентами системы ценообразования.",
  "explanation": "Определение зависимостей перед рефакторингом функции расчета скидок"
}
```

## Выходные данные

### Структура ответа

Агент возвращает структурированный анализ:

#### 1. Релевантные файлы
- Список файлов, связанных с проблемой
- Пути к файлам
- Краткое описание роли каждого файла

#### 2. Структура проекта
- Организация директорий
- Архитектурные паттерны
- Ключевые модули и их назначение

#### 3. Анализ зависимостей
- Как компоненты взаимодействуют
- Потоки данных
- Точки интеграции

#### 4. Рекомендации
- Какие файлы нужно изменить
- Порядок внесения изменений
- Потенциальные риски

### Пример выходных данных

```markdown
## Анализ системы регистрации

### Релевантные файлы:

1. **src/components/RegistrationForm.jsx**
   - Основной компонент формы регистрации
   - Обрабатывает пользовательский ввод
   - Вызывает API для отправки данных

2. **src/api/authService.js**
   - Сервис для работы с API аутентификации
   - Содержит функцию registerUser()
   - Обрабатывает HTTP запросы

3. **src/store/authSlice.js**
   - Redux slice для управления состоянием аутентификации
   - Содержит actions и reducers
   - Хранит статус регистрации

4. **src/utils/validation.js**
   - Утилиты валидации форм
   - Проверка email, пароля, и других полей

### Архитектура:

```
RegistrationForm → authService.registerUser() → API Backend
       ↓
   authSlice (Redux state management)
       ↓
   Validation utilities
```

### Зависимости:

- RegistrationForm зависит от authService и validation utils
- authService использует axios для HTTP запросов
- authSlice интегрирован с Redux store

### Рекомендации:

1. Начните с проверки authService.registerUser() - возможно проблема в HTTP запросе
2. Проверьте обработку ошибок в RegistrationForm
3. Убедитесь, что Redux middleware настроен корректно
4. Проверьте network requests в браузере для диагностики

### Потенциальные проблемы:

- Отсутствие обработки ошибок сети
- Неправильная конфигурация CORS
- Проблемы с валидацией на клиенте
```

## Когда использовать

### ✅ Используйте context-gatherer когда:

- Работаете с незнакомой кодовой базой
- Нужно понять, где находится функциональность
- Расследуете баг и не знаете, где искать
- Планируете большие изменения и нужно понять зависимости
- Нужно найти все использования определенной функции/класса
- Изучаете архитектуру проекта
- Не уверены, какие файлы релевантны для задачи

### ❌ НЕ используйте context-gatherer когда:

- Уже знаете конкретные файлы для изменения
- Работаете с маленьким проектом (< 10 файлов)
- Нужно просто прочитать один файл
- Задача не требует понимания архитектуры

---

## Ограничения

### Частота использования
- **Максимум один раз на запрос пользователя**
- Для дополнительного анализа используйте другие инструменты

### Размер проекта
- Оптимален для средних и больших проектов
- Для маленьких проектов может быть избыточным

### Глубина анализа
- Фокусируется на релевантных файлах
- Не анализирует весь проект целиком
- Использует эффективные стратегии поиска

---

## Стратегии использования

### Стратегия 1: Исследование перед изменениями

```
1. Вызов context-gatherer для понимания структуры
2. Анализ полученных результатов
3. Планирование изменений
4. Выполнение изменений через другие агенты
```

### Стратегия 2: Расследование бага

```
1. Вызов context-gatherer с описанием бага
2. Получение списка подозрительных файлов
3. Ручной анализ найденных файлов
4. Исправление через general-task-execution или bugfix-workflow
```

### Стратегия 3: Архитектурный анализ

```
1. Вызов context-gatherer с вопросом об архитектуре
2. Получение карты компонентов и зависимостей
3. Документирование архитектуры
4. Планирование рефакторинга
```

## Best Practices

### Формулирование эффективных запросов

✅ **Хорошо**:
```
"Как реализована система оплаты? Найди все файлы, связанные с обработкой 
платежей, интеграцией с payment gateway, и хранением транзакций."
```

❌ **Плохо**:
```
"Покажи мне код"
```

### Специфичность

Чем конкретнее запрос, тем точнее результат:
- Упоминайте конкретные функции/классы, если знаете
- Описывайте симптомы проблемы
- Указывайте область интереса (frontend, backend, database)

### Итеративный подход

Если первый анализ недостаточен:
1. Используйте полученную информацию для уточнения
2. Читайте найденные файлы вручную
3. Задавайте уточняющие вопросы пользователю

## Производительность

### Оптимизация времени выполнения

- Агент использует эффективные стратегии поиска
- Избегает чтения всех файлов проекта
- Фокусируется на релевантных областях
- Использует индексы и метаданные

### Ожидаемое время выполнения

- Маленький проект (< 50 файлов): 5-10 секунд
- Средний проект (50-500 файлов): 10-30 секунд
- Большой проект (> 500 файлов): 30-60 секунд

---

## Интеграция с workflow

### Типичная последовательность

```
context-gatherer → Анализ результатов → Планирование → Выполнение
```

### Пример полного workflow

1. **Сбор контекста**: context-gatherer анализирует проблему
2. **Планирование**: Создание спецификации через feature/bugfix workflow
3. **Реализация**: spec-task-execution выполняет задачи
4. **Дополнения**: general-task-execution для финальных штрихов

## Примеры реальных сценариев

### Сценарий 1: Новый разработчик в проекте

**Ситуация**: Новый разработчик присоединился к проекту и не знает структуру

**Вызов**:
```javascript
invokeSubAgent({
  name: "context-gatherer",
  prompt: "Это e-commerce приложение. Объясни общую архитектуру: как организованы компоненты, где находится бизнес-логика, как работает state management, и где API интеграция.",
  explanation: "Онбординг нового разработчика - обзор архитектуры проекта"
})
```

### Сценарий 2: Подготовка к рефакторингу

**Ситуация**: Нужно отрефакторить старый модуль, но неясны все зависимости

**Вызов**:
```javascript
invokeSubAgent({
  name: "context-gatherer",
  prompt: "Модуль src/legacy/userManager.js нужно отрефакторить. Найди все места, где он используется, какие функции вызываются чаще всего, и какие другие модули от него зависят.",
  explanation: "Анализ зависимостей перед рефакторингом legacy модуля"
})
```

### Сценарий 3: Расследование performance issue

**Ситуация**: Приложение медленно работает, но неясно где проблема

**Вызов**:
```javascript
invokeSubAgent({
  name: "context-gatherer",
  prompt: "Страница dashboard загружается медленно. Найди компоненты, которые рендерятся на dashboard, какие API запросы делаются, и где может быть узкое место.",
  explanation: "Идентификация компонентов для анализа производительности dashboard"
})
```
