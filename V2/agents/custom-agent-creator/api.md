# Custom Agent Creator - API Documentation

## Описание

Специализированный агент для создания и настройки новых пользовательских агентов в системе Kiro. Помогает расширить возможности системы через создание агентов под конкретные нужды.

---

## API Вызова

### Метод вызова
```
invokeSubAgent
```

### Обязательные параметры

#### name
- **Тип**: `string`
- **Значение**: `"custom-agent-creator"`
- **Описание**: Идентификатор агента

#### prompt
- **Тип**: `string`
- **Описание**: Описание требований к новому агенту
- **Формат**: Детальное описание назначения, возможностей и поведения агента

#### explanation
- **Тип**: `string`
- **Описание**: Объяснение необходимости создания нового агента
- **Формат**: Краткое обоснование

---

## Входные данные

### Структура prompt

Prompt должен включать:
- Назначение нового агента
- Специализация и область применения
- Требуемые возможности
- Ожидаемое поведение
- Интеграция с существующими системами

### Примеры входных данных

#### Пример 1: Агент для code review
```json
{
  "name": "custom-agent-creator",
  "prompt": "Создай агента для автоматического code review. Агент должен анализировать pull requests, проверять соответствие coding standards, находить потенциальные баги, и предлагать улучшения. Должен поддерживать JavaScript, TypeScript, и Python.",
  "explanation": "Создание специализированного агента для автоматизации code review процесса"
}
```

#### Пример 2: Агент для документации
```json
{
  "name": "custom-agent-creator",
  "prompt": "Создай агента, который генерирует API документацию из кода. Агент должен парсить JSDoc комментарии, TypeScript типы, и создавать markdown документацию в формате, совместимом с Docusaurus.",
  "explanation": "Автоматизация генерации API документации из исходного кода"
}
```

## Выходные данные

### Структура ответа

Агент создает:

#### 1. Конфигурационные файлы
- Файл конфигурации агента
- Настройки поведения
- Определение доступных инструментов

#### 2. Промпты и инструкции
- Системный промпт агента
- Правила поведения
- Примеры использования

#### 3. Интеграционные файлы
- Регистрация агента в системе
- Настройки активации
- Документация по использованию

### Пример выходных данных

```markdown
## Создан новый агент: code-reviewer

### Файлы:
- `.kiro/agents/code-reviewer/config.json` - Конфигурация
- `.kiro/agents/code-reviewer/prompt.md` - Системный промпт
- `.kiro/agents/code-reviewer/README.md` - Документация

### Возможности:
- Анализ кода на соответствие стандартам
- Обнаружение потенциальных багов
- Предложения по улучшению
- Поддержка JavaScript, TypeScript, Python

### Использование:
```javascript
invokeSubAgent({
  name: "code-reviewer",
  prompt: "Проверь pull request #123",
  explanation: "Автоматический code review"
})
```

### Активация:
Агент доступен через invokeSubAgent с name="code-reviewer"
```

## Когда использовать

### ✅ Используйте custom-agent-creator когда:

- Нужен агент для повторяющихся специализированных задач
- Существующие агенты не покрывают ваши нужды
- Хотите автоматизировать специфичный workflow
- Нужна кастомизация поведения для вашего проекта
- Создаете инструменты для команды

### ❌ НЕ используйте custom-agent-creator когда:

- Задача разовая (используйте general-task-execution)
- Существующий агент уже решает вашу проблему
- Задача слишком простая для отдельного агента

## Процесс создания агента

### Этап 1: Определение требований
- Агент собирает информацию о назначении
- Уточняет специализацию
- Определяет необходимые инструменты

### Этап 2: Проектирование
- Создание структуры агента
- Определение входных/выходных параметров
- Планирование поведения

### Этап 3: Реализация
- Создание конфигурационных файлов
- Написание промптов
- Настройка интеграции

### Этап 4: Тестирование
- Проверка работоспособности
- Тестовые вызовы
- Валидация результатов

---

## Best Practices

### Проектирование агента

1. **Четкая специализация**: Агент должен решать конкретную задачу хорошо
2. **Переиспользуемость**: Агент должен быть полезен в разных контекстах
3. **Документированность**: Четкое описание возможностей и ограничений
4. **Интеграция**: Совместимость с существующими агентами

### Формулирование требований

✅ **Хорошо**:
```
"Создай агента для миграции базы данных. Агент должен:
- Анализировать текущую схему БД
- Генерировать migration файлы
- Проверять безопасность миграций
- Создавать rollback скрипты
- Поддерживать PostgreSQL и MySQL"
```

❌ **Плохо**:
```
"Сделай агента для БД"
```

## Примеры специализированных агентов

### Пример 1: Test Generator Agent

**Назначение**: Автоматическая генерация unit тестов

**Вызов**:
```javascript
invokeSubAgent({
  name: "custom-agent-creator",
  prompt: "Создай агента test-generator, который анализирует функции и классы, и автоматически генерирует unit тесты. Должен поддерживать Jest и Vitest, создавать тесты для edge cases, и обеспечивать минимум 80% coverage.",
  explanation: "Автоматизация создания unit тестов для ускорения разработки"
})
```

### Пример 2: API Client Generator Agent

**Назначение**: Генерация клиентского кода из OpenAPI спецификации

**Вызов**:
```javascript
invokeSubAgent({
  name: "custom-agent-creator",
  prompt: "Создай агента api-client-generator, который читает OpenAPI 3.0 спецификацию и генерирует TypeScript клиент с типизированными методами для каждого endpoint. Должен включать error handling и retry logic.",
  explanation: "Автоматизация генерации API клиентов из спецификаций"
})
```

### Пример 3: Security Auditor Agent

**Назначение**: Аудит безопасности кода

**Вызов**:
```javascript
invokeSubAgent({
  name: "custom-agent-creator",
  prompt: "Создай агента security-auditor, который сканирует код на уязвимости: SQL injection, XSS, CSRF, insecure dependencies. Должен генерировать отчет с приоритетами и рекомендациями по исправлению.",
  explanation: "Автоматизация security аудита кодовой базы"
})
```

## Ограничения и соображения

### Технические ограничения
- Агент не может создавать агентов с доступом к новым инструментам
- Новый агент использует существующий набор инструментов
- Ограничения системы применяются ко всем агентам

### Поддержка и обслуживание
- Созданные агенты требуют тестирования
- Может потребоваться итеративная настройка
- Документация агента должна поддерживаться

## Интеграция с экосистемой

### Использование созданного агента

После создания агент доступен через:
```javascript
invokeSubAgent({
  name: "your-custom-agent-name",
  prompt: "Задача для агента",
  explanation: "Объяснение"
})
```

### Комбинирование агентов

Созданные агенты могут работать совместно с существующими:
```
context-gatherer → custom-agent → general-task-execution
```
