# General Task Execution Agent - API Documentation

## Описание

Универсальный агент общего назначения с полным доступом ко всем инструментам системы. Используется для выполнения произвольных задач разработки, которые не требуют специализированного подхода.

---

## API Вызова

### Метод вызова
```
invokeSubAgent
```

### Обязательные параметры

#### name
- **Тип**: `string`
- **Значение**: `"general-task-execution"`
- **Описание**: Идентификатор агента

#### prompt
- **Тип**: `string`
- **Описание**: Четкое описание задачи, которую должен выполнить агент
- **Формат**: Свободный текст с конкретными инструкциями

#### explanation
- **Тип**: `string`
- **Описание**: Одно-два предложения, объясняющие, почему используется этот агент и какая задача делегируется
- **Формат**: Краткое объяснение

### Опциональные параметры

#### preset
- **Тип**: `string | null`
- **Значение**: `null` (не используется для этого агента)
- **Описание**: Агент не использует пресеты

---

## Входные данные

### Структура prompt

Prompt должен содержать:
- Четкое описание задачи
- Контекст выполнения (если необходим)
- Ожидаемый результат
- Любые ограничения или требования

### Пример входных данных

```json
{
  "name": "general-task-execution",
  "prompt": "Создай утилитарную функцию для валидации email адресов. Функция должна проверять базовый формат email и возвращать boolean. Добавь unit тесты с использованием Jest.",
  "explanation": "Делегирование создания утилитарной функции с тестами универсальному агенту"
}
```

---

## Выходные данные

### Структура ответа

Агент возвращает результат выполнения задачи, который может включать:

- **Созданные файлы**: Список новых файлов с путями
- **Измененные файлы**: Список модифицированных файлов
- **Выполненные команды**: Команды, которые были запущены
- **Результаты тестирования**: Если тесты были запущены
- **Сообщения об ошибках**: Если возникли проблемы
- **Статус выполнения**: Успех или неудача

### Пример выходных данных

```json
{
  "status": "success",
  "filesCreated": [
    "src/utils/validateEmail.js",
    "src/utils/__tests__/validateEmail.test.js"
  ],
  "filesModified": [],
  "testsRun": true,
  "testResults": {
    "passed": 5,
    "failed": 0,
    "total": 5
  },
  "message": "Функция валидации email создана с полным покрытием тестами"
}
```

## Примеры использования

### Пример 1: Создание утилитарной функции

**Вызов**:
```javascript
invokeSubAgent({
  name: "general-task-execution",
  prompt: "Создай функцию debounce в src/utils/debounce.ts. Функция должна принимать callback и delay, возвращать debounced версию функции. Добавь TypeScript типы и JSDoc комментарии.",
  explanation: "Создание утилитарной функции debounce с типизацией"
})
```

**Результат**:
- Создан файл `src/utils/debounce.ts` с функцией
- Добавлены TypeScript типы
- Добавлены JSDoc комментарии
- Функция готова к использованию

---

### Пример 2: Рефакторинг кода

**Вызов**:
```javascript
invokeSubAgent({
  name: "general-task-execution",
  prompt: "Отрефактори файл src/components/UserList.jsx: извлеки логику фильтрации в отдельный хук useUserFilter, улучши читаемость кода, добавь PropTypes.",
  explanation: "Рефакторинг компонента с извлечением логики в кастомный хук"
})
```

**Результат**:
- Создан файл `src/hooks/useUserFilter.js`
- Обновлен `src/components/UserList.jsx`
- Добавлены PropTypes
- Улучшена структура кода

---

### Пример 3: Настройка конфигурации

**Вызов**:

invokeSubAgent({

### Пример 4: Исследование и анализ

## Возможности агента

### Доступные инструменты

Агент имеет доступ ко всем инструментам системы:
- Чтение и запись файлов
- Выполнение bash команд
- Поиск по коду
- Рефакторинг и редактирование
- Запуск тестов
- Диагностика ошибок
- Работа с git
- Веб-поиск и fetch

### Поддерживаемые задачи

- Создание новых файлов и модулей
- Модификация существующего кода
- Рефакторинг и оптимизация
- Написание тестов
- Настройка конфигураций
- Установка зависимостей
- Отладка и исправление ошибок
- Анализ кода
- Документирование

---

## Когда использовать

### Подходит для:
- Разовые задачи разработки
- Быстрые изменения кода
- Создание утилит и хелперов
- Настройка инструментов
- Исследовательские задачи
- Прототипирование

### Не подходит для:
- Создание спецификаций (используйте spec workflow агенты)
- Систематическое исправление багов (используйте bugfix-workflow)
- Разработка больших функций (используйте feature workflow агенты)
- Сбор контекста в незнакомом репозитории (используйте context-gatherer)

---

## Обработка ошибок

### Типичные ошибки

**Недостаточно контекста**:
- Агент запросит дополнительную информацию
- Предоставьте больше деталей в prompt

**Задача слишком сложная**:
- Разбейте на несколько подзадач
- Вызовите агента несколько раз последовательно

**Отсутствуют зависимости**:
- Агент попытается установить их
- Или сообщит о необходимости ручной установки

### Стратегии восстановления

1. Уточните prompt с большим количеством деталей
2. Разбейте задачу на более мелкие части
3. Предоставьте примеры ожидаемого результата
4. Укажите конкретные файлы для работы

## Best Practices

### Формулирование prompt

✅ **Хорошо**:
```
"Создай React компонент Button в src/components/Button.tsx. 
Компонент должен принимать props: label, onClick, disabled, variant (primary/secondary).
Используй styled-components для стилизации. Добавь storybook stories."
```

❌ **Плохо**:
```
"Сделай кнопку"
```

### Структура задачи

1. Что нужно создать/изменить
2. Где это должно находиться
3. Какие требования и ограничения
4. Какие дополнительные артефакты нужны (тесты, документация)

### Итеративный подход

Для сложных задач:
1. Первый вызов: Создание базовой структуры
2. Второй вызов: Добавление функциональности
3. Третий вызов: Тестирование и полировка

---

## Ограничения

- Агент автономен в рамках одной задачи
- Не сохраняет состояние между вызовами
- Каждый вызов - независимая сессия
- Для связанных задач нужно предоставлять контекст в каждом вызове

---

## Интеграция с другими агентами

### Последовательность использования

1. **context-gatherer** → Сбор информации о кодовой базе
2. **general-task-execution** → Выполнение конкретной задачи
3. **general-task-execution** → Дополнительные изменения (если нужно)

### Комбинирование с spec workflow

1. Создайте спецификацию через feature/bugfix workflow
2. Используйте spec-task-execution для задач из спецификации
3. Используйте general-task-execution для дополнительных задач вне спецификации
