# Документация по агентам Kiro

## Обзор

Kiro использует специализированных агентов для выполнения различных задач разработки. Каждый агент имеет свою область ответственности и активируется при определенных условиях.

---

## Доступные агенты

### 1. general-task-execution
**Назначение**: Универсальный агент для выполнения произвольных задач

**Когда активируется**:
- Общие задачи разработки, не требующие специализированного подхода
- Задачи, не связанные со спецификациями
- Произвольные запросы пользователя

**Входные параметры**:
- `prompt` (обязательный): Описание задачи для выполнения
- `explanation` (обязательный): Объяснение, почему используется этот агент

**Выходные данные**:
- Результат выполнения задачи
- Созданные/измененные файлы
- Отчет о выполненных действиях

**Доступ к инструментам**: Полный доступ ко всем инструментам системы

---

### 2. context-gatherer
**Назначение**: Анализ структуры репозитория и сбор контекста

**Когда активируется**:
- Необходимо понять структуру незнакомого кодовой базы
- Расследование багов в больших проектах
- Нужно определить, какие файлы релевантны для решения задачи
- Анализ взаимодействия компонентов

**Входные параметры**:
- `prompt` (обязательный): Описание проблемы или области для анализа
- `explanation` (обязательный): Объяснение цели сбора контекста

**Выходные данные**:
- Список релевантных файлов
- Структура проекта
- Анализ зависимостей между компонентами
- Рекомендации по дальнейшим действиям

**Ограничения**: 
- Использовать только один раз на запрос
- Не использовать, если уже известны конкретные файлы для изменения

---

### 3. custom-agent-creator
**Назначение**: Создание и настройка новых пользовательских агентов

**Когда активируется**:
- Пользователь запрашивает создание нового агента
- Необходима настройка специализированного агента

**Входные параметры**:
- `prompt` (обязательный): Описание требований к новому агенту
- `explanation` (обязательный): Объяснение необходимости создания агента

**Выходные данные**:
- Конфигурация нового агента
- Файлы настройки агента
- Инструкции по использованию

---

### 4. spec-task-execution
**Назначение**: Выполнение задач из спецификаций с поддержкой property-based testing

**Когда активируется**:
- Выполнение конкретной задачи из tasks.md
- Команды типа "execute task 2", "start task 1.3"
- Режим "run all tasks"

**Входные параметры**:
- `prompt` (обязательный): Детали задачи, путь к спецификации, контекст
- `explanation` (обязательный): Краткое описание делегируемой задачи
- Контекст включает:
  - ID и текст задачи
  - Путь к спецификации
  - Подзадачи (если есть)
  - Релевантный контекст из файлов спецификации

**Выходные данные**:
- Реализованный код
- Результаты тестирования
- Обновленный статус задачи
- Отчет о выполнении

**Особенности**:
- Поддержка property-based testing
- Автоматическое обновление статусов задач
- Обработка ошибок тестирования

---

### 5. feature-requirements-first-workflow
**Назначение**: Создание спецификации новой функциональности (подход "требования → дизайн → задачи")

**Когда активируется**:
- Пользователь выбрал "Build a Feature" и "Requirements"
- Есть четкие бизнес-требования, но нужен технический дизайн
- Традиционный подход разработки

**Входные параметры**:
- `name`: "feature-requirements-first-workflow"
- `preset`: Фаза workflow ("requirements", "design", "tasks")
- `prompt`: Исходный запрос пользователя с контекстом
- `explanation`: Причина вызова агента

**Выходные данные**:
- `requirements.md` - Документ требований
- `design.md` - Технический дизайн
- `tasks.md` - План реализации с задачами

**Процесс**:
1. Сбор и документирование требований
2. Создание технического дизайна на основе требований
3. Разработка плана реализации

---

### 6. feature-design-first-workflow
**Назначение**: Создание спецификации новой функциональности (подход "дизайн → требования → задачи")

**Когда активируется**:
- Пользователь выбрал "Build a Feature" и "Technical Design"
- Технический подход ясен, но нужно формализовать требования
- Рефакторинг или документирование существующей системы

**Входные параметры**:
- `name`: "feature-design-first-workflow"
- `preset`: Фаза workflow ("design", "requirements", "tasks")
- `prompt`: Исходный запрос пользователя с контекстом
- `explanation`: Причина вызова агента

**Выходные данные**:
- `design.md` - Технический дизайн
- `requirements.md` - Документ требований (выведенный из дизайна)
- `tasks.md` - План реализации с задачами

**Процесс**:
1. Создание технического дизайна
2. Формализация требований на основе дизайна
3. Разработка плана реализации

**Опции дизайна**:
- High-Level Design: Диаграммы системы, компоненты, модели данных
- Low-Level Design: Код/псевдокод, алгоритмы, сигнатуры функций

---

### 7. bugfix-workflow
**Назначение**: Систематическое исправление багов с использованием методологии bug condition

**Когда активируется**:
- Пользователь выбрал "Fix a Bug"
- Что-то сломано, падает или работает некорректно
- Обнаружена регрессия

**Входные параметры**:
- `name`: "bugfix-workflow"
- `preset`: Фаза workflow ("requirements", "design", "tasks")
- `prompt`: Описание бага с контекстом
- `explanation`: Причина вызова агента

**Выходные данные**:
- `bugfix.md` - Документ с описанием бага и условием бага C(X)
- `design.md` - План исправления
- `tasks.md` - Задачи для исправления

**Ключевые концепции**:
- Bug condition C(X): Формальное условие, определяющее баг
- Preservation checking: Проверка, что исправление не нарушает существующую функциональность
- Fix checking: Проверка, что исправление устраняет баг

**Процесс**:
1. Определение условия бага C(X)
2. Создание exploration test (должен упасть на багованном коде)
3. Разработка плана исправления
4. Реализация исправления
5. Проверка preservation и fix

## Правила активации агентов

### Оркестратор спецификаций

Главный оркестратор управляет выбором workflow и делегированием:

**Шаг 1: Выбор типа спецификации**
- Анализ запроса пользователя на индикаторы
- Индикаторы багфикса: "fix", "bug", "crash", "error", "broken", "issue", "problem"
- Индикаторы новой функции: "add", "new", "create", "implement", "build"
- Представление выбора пользователю

**Шаг 2: Выбор workflow (только для новых функций)**
- "Requirements" → feature-requirements-first-workflow
- "Technical Design" → feature-design-first-workflow
- Для багфиксов workflow выбирается автоматически (bugfix-workflow)

**Шаг 3: Делегирование**
- Вызов соответствующего агента с правильным preset
- Обработка завершения
- Предоставление следующих шагов пользователю

---

## Валидация предварительных условий

Перед делегированием к фазе tasks проверяются обязательные файлы:

**Requirements-first workflow**:
- Перед tasks → Проверка наличия requirements.md И design.md
- Если design.md отсутствует → делегирование к "design" preset
- Если requirements.md отсутствует → делегирование к "requirements" preset

**Design-first workflow**:
- Перед tasks → Проверка наличия design.md И requirements.md
- Если requirements.md отсутствует → делегирование к "requirements" preset
- Если design.md отсутствует → делегирование к "design" preset

**Bugfix workflow**:
- Перед tasks → Проверка наличия bugfix.md И design.md
- Если design.md отсутствует → делегирование к "design" preset
- Если bugfix.md отсутствует → делегирование к "requirements" preset

---

## Структура файлов спецификаций

Все файлы спецификаций следуют структуре:
- Директория функции: `.kiro/specs/{feature_name}/`
- Формат имени: kebab-case (например, "user-authentication")
- Обязательные файлы:
  - `requirements.md` или `bugfix.md` - Документ требований
  - `design.md` - Документ дизайна
  - `tasks.md` - Список задач реализации
  - `.config.kiro` - Конфигурация (specType, workflowType)

---

## Режимы выполнения задач

### Режим одной задачи
- Пользователь запрашивает выполнение конкретной задачи
- Оркестратор обновляет статус → делегирует spec-task-execution → обновляет статус

### Режим всех задач
- Пользователь запрашивает "run all tasks"
- Оркестратор:
  1. Читает tasks.md
  2. Помечает все незавершенные обязательные задачи как queued
  3. Последовательно делегирует каждую задачу spec-task-execution
  4. Обрабатывает ошибки и отчитывается о прогрессе

**Формат задач**:
- `- [ ]` = Не начата (обязательная)
- `- [x]` = Завершена
- `- [-]` = В процессе
- `- [~]` = В очереди
- `- [ ]*` = Опциональная задача (пропускается в режиме "run all")

## Обработка ошибок

### Ошибка вызова агента
- Логирование деталей ошибки
- Информирование пользователя
- Предложение альтернатив

### Ошибка создания контекста
- Создание минимального контекста
- Логирование для отладки
- Продолжение с уменьшенным контекстом

### Неверный выбор workflow
- Повторное представление опций с более четкими описаниями
- Примеры использования каждого workflow
- Принятие вариаций и маппинг к валидным опциям

---

## Критические правила

1. **Минимальность контекста**: Оркестратор хранит только логику выбора workflow
2. **Взаимодействие с пользователем**: Всегда использовать userInput для выбора workflow
3. **Протокол делегирования**: Всегда использовать invokeSubAgent для делегирования
4. **Последовательность**: НЕ вызывать параллельные агенты для создания спецификаций
5. **Прозрачность**: НЕ упоминать пользователю о существовании агентов и делегировании
6. **Только чтение для оркестратора**: Оркестратор НЕ пишет код, только координирует

## Примеры использования

### Пример 1: Создание новой функции
```
Пользователь: "Я хочу добавить аутентификацию пользователей"
→ Оркестратор: Представляет выбор типа спецификации
→ Пользователь: "Build a Feature"
→ Оркестратор: Представляет выбор workflow
→ Пользователь: "Requirements"
→ Оркестратор: Делегирует feature-requirements-first-workflow
→ Результат: Создана спецификация в .kiro/specs/user-authentication/
```

### Пример 2: Исправление бага
```
Пользователь: "Приложение падает когда количество равно нулю"
→ Оркестратор: Представляет выбор типа спецификации (рекомендует Bugfix)
→ Пользователь: "Fix a Bug"
→ Оркестратор: Делегирует bugfix-workflow (пропускает выбор workflow)
→ Результат: Создана спецификация багфикса в .kiro/specs/quantity-zero-crash-fix/
```

### Пример 3: Выполнение задачи
```
Пользователь: "execute task 2"
→ Оркестратор: Читает спецификацию
→ Оркестратор: Обновляет статус задачи на in_progress
→ Оркестратор: Делегирует spec-task-execution
→ spec-task-execution: Реализует задачу, запускает тесты
→ Оркестратор: Обновляет статус задачи на completed
```
