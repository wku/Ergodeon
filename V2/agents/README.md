# Kiro Agents - API Documentation

Полная документация по всем агентам системы Kiro с примерами использования, входными/выходными параметрами и best practices.

---

## Структура документации

Каждый агент имеет свою папку с файлом `api.md`, содержащим:
- Описание и назначение
- API вызова с параметрами
- Примеры входных данных
- Структуру выходных данных
- Примеры использования
- Best practices
- Ограничения и troubleshooting

---

## Список агентов

### 1. General Task Execution
**Папка**: `agents/general-task-execution/`  
**Назначение**: Универсальный агент для произвольных задач разработки  
**Когда использовать**: Разовые задачи, не требующие специализированного подхода

### 2. Context Gatherer
**Папка**: `agents/context-gatherer/`  
**Назначение**: Анализ структуры репозитория и сбор контекста  
**Когда использовать**: Незнакомая кодовая база, расследование багов, понимание архитектуры

### 3. Spec Task Execution
**Папка**: `agents/spec-task-execution/`  
**Назначение**: Выполнение задач из спецификаций с PBT поддержкой  
**Когда использовать**: Реализация задач из tasks.md файлов

### 4. Custom Agent Creator
**Папка**: `agents/custom-agent-creator/`  
**Назначение**: Создание новых пользовательских агентов  
**Когда использовать**: Нужен специализированный агент для повторяющихся задач

### 5. Feature Requirements-First Workflow
**Папка**: `agents/feature-requirements-first-workflow/`  
**Назначение**: Создание спецификаций (требования → дизайн → задачи)  
**Когда использовать**: Бизнес-требования определяют технический дизайн

### 6. Feature Design-First Workflow
**Папка**: `agents/feature-design-first-workflow/`  
**Назначение**: Создание спецификаций (дизайн → требования → задачи)  
**Когда использовать**: Технический подход ясен, нужно формализовать требования

### 7. Bugfix Workflow
**Папка**: `agents/bugfix-workflow/`  
**Назначение**: Систематическое исправление багов с bug condition методологией  
**Когда использовать**: Что-то сломано и требует формального исправления

### Выполнить простую задачу
```javascript

  name: "general-task-execution",
  prompt: "Создай компонент Button в React",
```

## Архитектура агентов

### Оркестратор

Главный оркестратор управляет:
- Выбором типа спецификации (Feature vs Bugfix)
- Выбором workflow (Requirements-first vs Design-first)
- Делегированием к специализированным агентам
- Валидацией предварительных условий

### Специализированные агенты

Каждый агент:
- Фокусируется на конкретной задаче
- Имеет четкие входы и выходы
- Работает автономно
- Возвращает структурированные результаты

### Взаимодействие агентов

```
Оркестратор
    ↓
    ├─→ context-gatherer (анализ)
    ├─→ feature-requirements-first-workflow (спецификация)
    ├─→ feature-design-first-workflow (спецификация)
    ├─→ bugfix-workflow (спецификация)
    ├─→ spec-task-execution (реализация)
    ├─→ general-task-execution (произвольные задачи)
    └─→ custom-agent-creator (расширение)
```

## Типичные сценарии использования

### Сценарий 1: Новая функция с нуля

```
1. Оркестратор → Выбор "Build a Feature"
2. Оркестратор → Выбор "Requirements"
3. feature-requirements-first-workflow → Создание requirements.md
4. [Одобрение пользователя]
5. feature-requirements-first-workflow → Создание design.md
6. [Одобрение пользователя]
7. feature-requirements-first-workflow → Создание tasks.md
8. spec-task-execution → Выполнение задач
```

### Сценарий 2: Исправление бага

```
1. Оркестратор → Выбор "Fix a Bug"
2. bugfix-workflow → Создание bugfix.md
3. [Одобрение пользователя]
4. bugfix-workflow → Создание design.md
5. [Одобрение пользователя]
6. bugfix-workflow → Создание tasks.md
7. spec-task-execution → Task 1 (exploration test)
8. spec-task-execution → Task 2 (fix implementation)
9. spec-task-execution → Task 3 (preservation checking)
10. spec-task-execution → Task 4 (fix checking)
```

### Сценарий 3: Быстрая задача без спецификации

```
1. general-task-execution → Выполнение задачи напрямую
```

### Сценарий 4: Исследование незнакомого кода

```
1. context-gatherer → Анализ структуры
2. [Анализ результатов]
3. general-task-execution или spec workflow → Реализация изменений
```

## Выбор правильного агента

### Блок-схема выбора

```
Нужна спецификация?
├─ Нет → general-task-execution
└─ Да → Это баг или новая функция?
    ├─ Баг → bugfix-workflow
    └─ Функция → Требования или дизайн первым?
        ├─ Требования → feature-requirements-first-workflow
        └─ Дизайн → feature-design-first-workflow

Нужен анализ кодовой базы?
└─ Да → context-gatherer (затем другой агент)

Нужен новый агент?
└─ Да → custom-agent-creator
```
